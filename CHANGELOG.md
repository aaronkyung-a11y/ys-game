## v0.53 (2026-07-15) — 앱 아이콘 교체

**기존**: 파이썬으로 그린 단순 도형 골든 검. 게임 화풍과 무관.

**신규**: 제미나이 생성 — 주인공(칼 든 정면)을 중심으로 뒤에 황금 기사와 오크가 실루엣으로 겹친 구도. 16비트 SNES 픽셀아트, 진한 남색 배경 + 황금 광채.

**아이콘 전용 프롬프트 요건** (일반 스프라이트와 다름)
- 배경을 **마젠타가 아니라 진한 남색(#0d0f14)으로 채움** — 아이콘은 투명하면 안 됨.
- **중앙 70% 안에 중요 요소 배치** — 런처가 원형/둥근사각으로 크롭하므로 모서리 콘텐츠는 잘림.
- **48x48에서 읽히도록** 굵고 단순하게 — 실제 홈 화면 표시 크기 기준.

**검증**: 안전 원 밖 콘텐츠 6.6%뿐, 네 모서리 전부 배경색(11,15,24) — 원형 크롭 통과.

**manifest 수정**: `purpose: "any maskable"` → `"any"`. maskable은 런처가 더 공격적으로 크롭하는데 이 아트는 그만큼의 패딩이 없어 잘릴 위험이 있었음.

## v0.52 (2026-07-15) — 타이틀 화면 · 저장/이어하기 · 클리어 화면

**문제**: 새로고침하거나 앱을 전환하면 진행이 전부 날아감. 테스트도 불편했음.

**저장 시스템**
- `localStorage`에 진행 저장 (`ys_save_v1`): 현재 맵, 레벨/경험치, maxHP, 공격 보너스, 이동속도, 클리어한 맵 목록.
- **자동 저장 시점**: 맵 이동 시, 레벨업 시, 게이트 개방 시.
- **적 위치는 저장하지 않음** — 전투 도중 저장/복원(세이브 스커밍)을 막고, 이어하기 시 해당 맵을 처음부터 시작.
- 이어하기 시 HP는 최대치로 복원 (맵 시작 지점이므로).

**타이틀 화면**
- 게임 상태 머신 도입: `title` / `playing` / `cleared`. 메뉴 중에도 루프는 돌아서 배경이 계속 움직임.
- 세이브가 있으면 **이어하기** 버튼 표시 (맵 이름 + 레벨 미리보기), 없으면 시작만.
- 메뉴에서는 HUD·스틱·버튼 숨김 (`setUIVisible`).

**클리어 화면**
- 황금 기사 처치 시 통계 표시: **레벨 · 소요 시간 · 사망 횟수**.
- 런 완료 시 세이브 삭제 (반쯤 끝난 세이브가 남지 않도록).
- 기존에는 최종 보스 처치 후 들판으로 무한 순환했으나, 이제 명확한 엔딩이 있음.

**검증**: 타이틀→새게임→진행→자동저장→**새로고침**→이어하기로 유적/Lv4/maxHP136/공격+2 완전 복원 확인. 클리어 화면→통계→다시하기 초기화 확인.

## v0.51 (2026-07-15) — 고블린 공격 방향 + 여기사 크기 일관성

**1. 고블린 공격 시 좌우 반전**
- 측정: 공격 행 단검이 left칸 **+43**(오른쪽), right칸 −43 — 걷기 행(−20/+19)과 **부호가 반대**였음.
- 원인: `build_sheets.py`의 `MANUAL_FLIP`에 있던 `('GOBLIN', 2)` 항목. v0.37에 넣었던 보정인데, 이후 파이프라인이 바뀌면서 **재빌드할 때마다 멀쩡한 행을 다시 뒤집고** 있었음.
- 수정: 해당 override 제거 + 현재 시트 스왑. 3개 행 전부 부호 일치 확인 (left 전부 음수).

**2. 여기사가 위/아래 공격 시 작아짐**
- 측정: 공격 행 몸 높이 down=61 up=58 (idle은 88~91) — **35% 축소**.
- 원인: `to_cell()`이 **전체 아트 높이**를 셀에 맞춤. UP/DOWN 공격은 검이 세로로 길어서 전체 높이가 크고, 그걸 맞추니 몸이 쪼그라듦. (가로 스트립인 LEFT/RIGHT는 덜 심했음)
- 수정: `to_cell_body()` 신설 — **몸통 bbox**(얇은 검날 제외, 두꺼운 행/열만)를 기준으로 스케일하고 몸통 중심으로 정렬. 검이 셀을 넘지 않는 선에서 상한.
- 결과: 12칸 몸 높이 **77~92px (편차 15px)**, 기존 58~97(편차 39px)에서 개선.

## v0.50 (2026-07-15) — 최종 보스: 황금 기사 + 4번째 맵

**진행 확장**: 들판 → 유적 → 오크 군주의 둥지 → **황금 기사의 홀** (1500x1200, 결투장). 오크 군주 처치 시 게이트가 열림.

**황금 기사 (최종 보스)**
- HP 130, XP 200, 접촉 14, 반지름 34. 렌더 190px.
- **오크 군주의 정반대**: 속도 3.4 (오크 1.3). 오크가 "예고를 읽고 피하는" 상대라면, 이쪽은 "반응해서 피하는" 상대.
- 패턴 4종:
  - **blink** — 사라졌다가 플레이어 옆 120~160px에 재출현 (반투명 페이드로 표현)
  - **thrust/lunge** — 고속 찌르기를 2연타 (격노 시 3연타)
  - **sweep** — 회전베기, 방사형 10발 (격노 14발). 밀착 응징
  - **beam** — 3갈래 검기 발사. 거리를 둬도 안전하지 않음
- **격노 (HP 45% 이하)**: 돌진 10→13, 연타 2→3, 패턴 간격 단축, 황금 틴트.
- 보스 HP바에 이름·격노 표시 (황금색), 미니맵 표시.

**스프라이트 제작 — 방법을 바꿔서 해결**
- 문제: 얇고 긴 검이 정사각 셀에 안 들어감. 제미나이에 "작게 그려라"를 **4번** 요청했으나 매번 캐릭터를 프레임에 꽉 채워 검이 잘림.
- 해결: **방향별로 캔버스 비율을 바꿔서 개별 생성**
  - RIGHT → 1024x377 가로 스트립 (검 끝 뒤 여백 469px)
  - UP → 377x1024 세로 스트립 (여백 413px)
  - DOWN → 377x1024 세로 스트립 (여백 448px)
  - LEFT → RIGHT 반전
  - idle 8칸 → 기존 12칸 시트에서 (검이 아래를 향해 여백 문제 없음, 측정 확인)
- 핵심: **픽셀 크기를 직접 지정**하고 "정사각형으로 만들지 마라" + "검 방향 절반을 비워라"를 명시. 비율만 말하면 무시당함.
- 검증: 12칸 전부 가장자리 접촉 0px.

**검증**: 둥지→홀 진행, 기사 스폰, blink/thrust/lunge/sweep/beam 전 패턴 동작, 회전베기 10→격노 14발, 검기 3발 확인.

## v0.49 (2026-07-15) — 공격 시 무기 잘림 수정

**증상**: 공격 자세에서 칼/도끼가 셀 밖으로 잘림.

**원인 (파이프라인 버그 2개)**
1. `to_cell()`이 **높이 기준으로만** 스케일 — 무기를 옆으로 뻗은 공격 자세는 세로보다 가로가 긴데, 세로에 맞춰 키우니 무기가 셀 폭을 넘음.
2. 더 근본적인 문제: x 위치를 **발 기준**으로 정렬한 뒤 셀 안으로 clamp. 공격 자세는 무기가 발에서 멀리 떨어져 있어 발 기준 x가 여유 공간을 초과 → clamp가 그림을 셀 가장자리에 **밀착**시킴.
   - 측정 예시(고블린 공격): 최종 111px 그림에 여유 17px인데 발 기준 x는 29 → 17로 잘려서 오른쪽 끝에 붙음.

**수정**
- 가로·세로 양축으로 스케일 계산 후 **작은 쪽 채택** (가로 12% 여백).
- x 앵커: **발 기준이 셀에 들어가면 사용, 안 들어가면 그림 자체를 중앙 정렬**. 걷기는 발 정렬 유지(프레임 간 미끄러짐 방지), 공격만 중앙 폴백.
- 오크 스케일 0.92 → 0.84 (도끼가 실제로 셀을 넘었음). 렌더 크기로 보정: charger 132→145, boss 250→274.

**검증**: 3개 시트 × 12칸 전부 **가장자리 접촉 0px** (기존 고블린 8px, 오크 41px).

## v0.48 (2026-07-15) — 보스 난이도 상향

**측정으로 드러난 문제**
- 들판+유적을 클리어하면 총 187 XP → 보스 도달 시 **Lv7, 공격 +3**.
- 보스 HP 60 / 풀콤보 16 데미지 → **3.8콤보(약 10초)면 사망**. 관문이 아니라 그냥 큰 잡몹이었음.

**공격력 보너스 스케일링 (근본 문제)**
- 기존: `dmg = sw.dmg + atkBonus` — **정액 가산**.
  - Lv7 기준 1타(1→4)는 **300% 증가**, 3타 피니셔(4→7)는 75%만 증가.
  - 결과: 콤보를 이을 이유가 사라짐. 1타 연타가 더 효율적이었음.
- 변경: `dmg = sw.dmg + round(atkBonus * (sw.dmg/2))` — **타격 무게에 비례**.
  - Lv7: 1타 3 / 2타 5 / 3타 **10**. 피니셔가 확실히 무겁게 유지됨.
  - 잡몹 TTK는 유지 (그런트·투척꾼 1타, 오크 2타).

**보스 스탯**
- HP 60 → **150**, 접촉 데미지 16 → **18**, XP 60 → **120** (제대로 된 전투가 됐으니 보상도 상향).

**공격성**
- `rest` 시간 단축: 슬램 후 44→30 (격노 18), 소환 후 50→34 (격노 20). 기존엔 대부분의 시간을 가만히 서 있었음.
- 격노 시 이동속도 1.5배.
- **격노 연쇄 돌진 (신규)**: 격노 페이즈에서 돌진을 최대 3연타로 이음 (`chain`). 한 번 피했다고 끝이 아님.

**검증**
- 회피 없이 계속 붙어서 때리면 25초에 플레이어 172→4 HP, 보스는 42만 깎임 — 이제 맞고 버틸 수 없음.
- 완벽 회피 가정 시 11초 클리어. 실전에선 회피하며 싸워야 하므로 30초~1분 예상.
- 격노 시 연쇄 돌진 chain=2(3연타) 확인, 전 패턴(charge/windup/slam/summon/rest) 정상.

## v0.46 (2026-07-15) — 베기 다듬기 (더 다이나믹하게)

**칼날 궤적에 이징 (핵심)**
- 기존: `prog = 1-(atkT/total)` — **선형**. 칼이 처음부터 끝까지 일정한 속도로 쓸고 지나감. 이게 "밋밋한" 근본 원인이었음.
- 변경: cubic ease-in-out 적용. 윈드업에선 칼이 뒤에 머물고, 타격 순간 **휙 채찍처럼** 지나간 뒤, 마무리에서 감속.
- 측정: 스윙 중간 30% 구간이 전체 호의 **66%**를 커버 (기존 균등 30%), 시작 30%는 11%만 커버.

**칼끝 광점**
- 궤적 선단에 밝은 방사형 광점 추가. 눈이 이 점을 추적하므로 **속도감을 만드는 건 실제로 이것**.
- 콤보 3타(gold)에서는 더 크고 황금색.

**전진(lunge) 프론트로딩**
- 기존: 액티브 프레임 내내 균등한 힘 (0.35 고정) — 스윙 모든 프레임이 똑같이 느껴짐.
- 변경: `0.35 + 0.9*t²` — 칼이 휘둘러지는 순간 몸이 **확 따라 들어가고** 이후 관성으로 흐름.

**타격감**
- 스파크가 속도 방향으로 **줄무늬**로 그려짐 (점 → 선). 정지된 점보다 속도가 읽힘.
- 베기 방향과 **수직으로 튀는 흰 파편** 추가 — "잘렸다"는 느낌.
- 접촉점에 짧은 **임팩트 플래시** (6~9프레임 방사형 그라디언트).
- 스파크에 감속(0.88/frame) 적용 — 등속으로 영원히 날아가지 않음.
- **데미지에 비례한 피드백**: 3+ 데미지나 처치 시 파티클 수 증가(10→23 확인), 히트스톱·화면흔들림 +3.

**성능**: 60 FPS 유지.

## v0.45 (2026-07-15) — 틴트 사각형 버그 수정 (피격/대시 시 네모칸)

**증상**: 맞을 때, 대시할 때 캐릭터 주변에 밝은 네모칸이 나타남.

**원인 (내 코드의 오래된 버그)**
- 모든 틴트 코드가 메인 캔버스에 직접 `globalCompositeOperation='source-atop'` + `fillRect(전체 셀 영역)`을 적용.
- `source-atop`은 **캔버스에 이미 그려진 모든 것**(바닥, 벽)을 대상으로 합성함 — 스프라이트 픽셀만이 아님.
- 결과: 스프라이트의 128px 셀 전체가 색칠됨 = 네모칸.
- **v0.44 이전에는 안 보였음** — 바닥이 거의 검정(#0d0f14)이라 어두운 틴트가 묻혔기 때문. 타일 바닥이 들어오면서 드러난 것.
- 영향 범위: 주인공(피격 백색/대시 청색), 적 3종(피격), 오크 돌진 틴트, 보스 격노 틴트 — 5개 함수 전부.

**수정**
- `drawSheetTinted()` 헬퍼 신설 — 오프스크린 스크래치 캔버스에 스프라이트만 그린 뒤 거기서 틴트 합성, 결과를 메인 캔버스에 blit.
  - 스크래치 캔버스에는 스프라이트만 있으므로 `source-atop`이 의도대로 동작.
  - 버퍼는 필요 시에만 확장하며 재사용 (매 프레임 캔버스 생성 안 함).
- `drawHero` / `drawGoblin` / `drawOrc` / `drawThrower` / `drawSwing` 5개 전부 이 헬퍼로 통합.

**검증**: 스프라이트 셀 모서리의 바닥 픽셀이 무틴트/피격/대시 세 상태에서 모두 [36,48,32]로 동일 — 사각형 소멸. 백색 플래시는 스프라이트에 정상 적용 확인 (박스 평균 [49,50,33] → [56,59,44]). FPS 60 유지.

**남은 개선 여지**: 근본 해결은 걷기 전용 2프레임(또는 4프레임)을 제미나이로 뽑는 것. 현재는 idle/walk 셀 재활용이라 스텝이 정석적이진 않음.

## v0.36 (2026-07-15) — 마젠타 잔여물 제거 + 시트 재빌드 (방향 행별 판정)

**마젠타 잔여물 제거 (확실히 해결)**
- 원인: flood-fill을 **테두리에서만** 수행 → 겨드랑이, 팔과 몸통 사이처럼 **사방이 막힌 주머니**의 마젠타는 테두리에서 도달 불가 → 그대로 남음.

- 수정: (1) 테두리 flood-fill → (2) 남은 마젠타 전역 스윕 → (3) 반투명 마젠타 프린지 제거.
  - 전역 스윕이 안전한 이유: 캐릭터 팔레트(녹색/회녹색/갈색/강철색)에 마젠타 계열이 없음.
- 결과: **3개 시트 전부 잔여 마젠타 0px**.

**시트 재빌드 — 방향을 행별로 측정**
- v0.34/v0.35 실패의 근본 원인: "제미나이 원본의 열 방향은 일정하다"는 가정. 실제로는 **시트마다, 심지어 같은 시트의 행마다 다름**.

  - 오크: col4가 left (뒤집기 필요)

- 수정: 노란 눈/붉은 눈 위치로 각 행의 실제 방향을 측정 → 필요한 행만 반전. `build_sheets.py`로 파이프라인 스크립트화하여 레포에 보관.

**미해결 / 검증 한계**
- 이 세션에서 이미지 뷰어가 계속 빈 결과를 반환해 **렌더 결과를 눈으로 확인하지 못함**.

## v0.34 (2026-07-15) — 적 좌우 반전 버그 수정 (뒤돌아보며 다가오던 문제)

- 교훈: 스프라이트 방향은 휴리스틱 대신 **렌더 결과를 직접 확인**할 것.

**수정**
- 3개 시트 모두 left/right 열을 스왑.
- 게임을 실제 렌더해서 검증: 플레이어 오른쪽에 적 배치 → `face='left'` + **화면상으로도 왼쪽을 봄** 확인.

## v0.29 (2026-07-15) — BGM 추가 (Skybound Caravan)

**음원 처리**
- 원본 6.15MB / 200kbps / 4분 18초 → **96kbps로 재인코딩 (2.9MB, 절반 이하)**. 게임 BGM엔 200kbps가 과함.
- 원본 끝부분이 페이드아웃돼 있어서 그냥 loop하면 소리가 사라졌다 갑자기 커지는 이음새가 생김 (끝 -38.6dB vs 시작 -27.5dB).
  → 페이드아웃 꼬리를 잘라내고(254초) 짧은 페이드인만 유지 → 이음새 -24.9dB / -29.8dB로 정합.

**임베드 방식 (중요)**
- base64로 index.html에 박지 **않음**. 6MB가 base64로 8MB가 되고, v0.27에서 HTML을 network-first로 바꿨기 때문에 **매 로드마다 8MB를 새로 받게 됨** — 모바일에서 치명적.
- 대신 `bgm.mp3` 별도 파일 + 서비스워커 정적 캐시(cache-first)에 등록. 첫 로드 후엔 오프라인에서도 재생되고 재다운로드 없음.

**재생 로직**
- 브라우저 자동재생 정책 때문에 로드 시 `play()`는 실패함 → 첫 탭/키 입력(user gesture)에서 재생 시작.
- 우상단 ♪ 버튼으로 on/off. 설정은 `localStorage`에 저장되어 다음 실행에도 유지.
- 탭/앱이 백그라운드로 가면 자동 일시정지, 돌아오면 재개.
- 볼륨 0.45 기본.

## v0.27 (2026-07-15) — 서비스워커 캐시 버그 수정 (버전이 안 올라가던 문제)

**증상**: 레포에는 v0.26이 올라가 있는데 브라우저/앱에서는 계속 v0.25가 뜸.

**원인 (내 설계 실수)**
- `sw.js`의 fetch 핸들러가 **cache-first**였음: `caches.match(req).then(hit => hit || fetch(req))`.
- 게임 전체가 `index.html` 한 파일에 들어있는데, 캐시에 index.html이 있으면 네트워크를 아예 조회하지 않음.
- 캐시 이름(`ys-v0_26`)을 올려도 브라우저가 **sw.js 자체를 캐시에서** 받으면 새 워커가 설치되지 않아 영원히 구버전에 잠김.
- 즉 "캐시 이름만 올리면 갱신된다"는 건 틀렸음.

**수정**
- HTML은 **network-first**로 전환: 항상 네트워크를 먼저 시도하고(`cache: 'no-store'`), 실패할 때만 캐시로 폴백. 오프라인 동작은 유지.
- 아이콘·매니페스트 등 정적 자산은 cache-first 유지 (자주 안 바뀜).
- 페이지 쪽에 자동 갱신 로직 추가: 로드 시 + 60초마다 `reg.update()`, 새 워커 감지되면 `SKIP_WAITING` → `controllerchange`에서 1회 리로드.

**결과**: 앞으로는 새로고침만 하면 최신 빌드가 뜸. 앱 삭제/재설치 불필요.
