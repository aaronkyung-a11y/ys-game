## v0.38 (2026-07-15) — 그런트 공격 방향 실제 수정 (v0.37 실패 원인 규명)

**v0.37이 왜 안 먹혔나**
- v0.37에서 공격 행을 스왑했지만 증상 동일. 배포는 정상이었음 (라이브 index.html의 시트 md5가 로컬과 일치, 버전도 v0.37).
- 실제 원인: **문제가 좌우 반전이 아니었음.** 파이프라인이 LEFT 칸을 항상 "col4를 반전"해서 만들고 **제미나이의 col3를 버리는** 구조였는데, 공격 행에서는 이 방식 자체가 부적절.
- 측정: 원본 공격 행의 단검 위치가 **col3 = -116(왼쪽), col4 = +115(오른쪽)** — 제미나이가 이미 양방향을 제대로 그려줬음. 걷기 행(-46/+46)보다 훨씬 뚜렷.
- 즉 멀쩡한 col3를 버리고 col4 반전본을 쓰면서, 공격 자세 특유의 몸통 비틀림·칼 궤적이 어긋난 것. 스왑해도 반전본끼리 바뀔 뿐이라 증상이 그대로였음.

**수정**
- 공격 행(3행)은 **제미나이 원본 col3/col4를 그대로 사용** (반전 없음).
- 걷기 행(1·2행)은 기존 방식 유지 — 검증 완료 상태.
- 검증: 최종 시트에서 단검 오프셋 좌열 -49 / 우열 +47 — 3개 행 전부 방향 일치.
- 오크·투척꾼 md5 무변경 확인 (사용자 승인 상태 보존).

**교훈**: "LEFT는 항상 RIGHT 반전"이라는 파이프라인 규칙이 걷기에는 맞지만 비대칭적인 액션 자세에는 맞지 않음. 원본이 양방향을 제대로 그렸다면 그걸 쓰는 게 맞음.

## v0.37 (2026-07-15) — 그런트 공격 방향 수정

**증상**: 그런트의 걷기는 정상인데 **공격할 때만** 방향이 반대 (사용자 확인).

**원인**
- 자동 방향 판정이 "눈은 바라보는 쪽에 있다"는 규칙을 쓰는데, 이는 **걷기 자세에서만 성립**.
- 공격 자세는 몸이 비틀리고 칼이 앞으로 뻗어 나가면서 얼굴 각도가 달라져 규칙이 깨짐.
- 실제로 원본 측정 시 고블린만 3행에서 `col1=right`로 다른 행과 다르게 튀었음 — 신호가 있었는데 넘어갔음.

**수정**
- 고블린 3행(공격)만 left/right 스왑. 걷기 2개 행과 오크·투척꾼은 **손대지 않음** (사용자 확인 완료 상태).
- `build_sheets.py`에 `MANUAL_FLIP` 테이블 추가: 자동 판정이 실패하는 행을 명시적으로 기록.
  - `('GOBLIN', 2)` — 공격 행
  - `('ORC', 2)` — 돌진 행 (도끼를 뒤로 젖히는 자세라 무기 위치 규칙이 반전됨)
- 이제 스크립트 재실행 시 승인된 시트가 그대로 재현됨 (md5 일치 확인).

**푸시 전 잡은 사고**
- 스크립트 재빌드 시 오크 시트가 v0.36과 **달라짐**을 발견. v0.36의 오크에는 스크립트 밖에서 수동으로 넣은 3행 스왑이 있었는데 재빌드가 이를 날려버림.
- 승인된 오크를 복원하고 해당 override를 스크립트에 기록하여 해결. 검증 없이 푸시했다면 또 오크를 망가뜨렸을 것.

## v0.36 (2026-07-15) — 마젠타 잔여물 제거 + 시트 재빌드 (방향 행별 판정)

**마젠타 잔여물 제거 (확실히 해결)**
- 원인: flood-fill을 **테두리에서만** 수행 → 겨드랑이, 팔과 몸통 사이처럼 **사방이 막힌 주머니**의 마젠타는 테두리에서 도달 불가 → 그대로 남음.
- 측정: 고블린 2000px, 오크 1947px, 투척꾼 5438px 잔존.
- 수정: (1) 테두리 flood-fill → (2) 남은 마젠타 전역 스윕 → (3) 반투명 마젠타 프린지 제거.
  - 전역 스윕이 안전한 이유: 캐릭터 팔레트(녹색/회녹색/갈색/강철색)에 마젠타 계열이 없음.
- 결과: **3개 시트 전부 잔여 마젠타 0px**.

**시트 재빌드 — 방향을 행별로 측정**
- v0.34/v0.35 실패의 근본 원인: "제미나이 원본의 열 방향은 일정하다"는 가정. 실제로는 **시트마다, 심지어 같은 시트의 행마다 다름**.
  - 고블린: 3행 전부 col4=right
  - 오크: col4가 left (뒤집기 필요)
  - 투척꾼: 1·2행 col4=right, **3행만 col4=left**
- 수정: 노란 눈/붉은 눈 위치로 각 행의 실제 방향을 측정 → 필요한 행만 반전. `build_sheets.py`로 파이프라인 스크립트화하여 레포에 보관.

**미해결 / 검증 한계**
- 이 세션에서 이미지 뷰어가 계속 빈 결과를 반환해 **렌더 결과를 눈으로 확인하지 못함**.
- 오크는 눈이 작아 자동 판정이 불안정했고, 도끼 위치/머리 질량/붉은 눈 세 가지 휴리스틱이 **서로 다른 답**을 냄 → 오크 방향은 미검증 상태.
- 고블린·투척꾼은 눈 위치 기준 9행 전부 판정 일치.
- 사용자 육안 확인 필요.

## v0.34 (2026-07-15) — 적 좌우 반전 버그 수정 (뒤돌아보며 다가오던 문제)

**증상**: 그런트가 플레이어를 등지고 다가옴. 공격 모션도 반대 방향.

**원인 (내 실수)**
- 시트 조립 시 "원본 4열(RIGHT)은 오른쪽을 본다"고 **가정**하고, 그걸 반전해서 LEFT 칸을 만들었음.
- 실제로는 제미나이 원본의 4열이 **이미 왼쪽을 보고 있었음**. 결과적으로 left/right 두 칸이 모두 반대로 들어감.
- 고블린·오크·투척꾼 3종 전부 같은 파이프라인이라 동일하게 오염됨.
- v0.33에서 AI 방향 로직(`dir4`, face 판정)은 이미 정상이었음 — `face='left'`가 나오는데 **그림이 오른쪽을 보고 있던 것**. 상태는 맞고 이미지가 틀린 케이스.

**디버깅 중 자책할 점**
- 방향을 자동 판정하려고 머리 부분 픽셀 질량을 좌우 비교했는데, 고블린은 **코가 앞으로 나오지만 귀가 뒤로 젖혀져** 있어서 귀 쪽 질량이 더 커 판정이 거꾸로 나옴. 측정값을 믿고 "시트 정상"이라 결론 냈다가, 실제 셀을 눈으로 보고서야 발견.
- 교훈: 스프라이트 방향은 휴리스틱 대신 **렌더 결과를 직접 확인**할 것.

**수정**
- 3개 시트 모두 left/right 열을 스왑.
- 게임을 실제 렌더해서 검증: 플레이어 오른쪽에 적 배치 → `face='left'` + **화면상으로도 왼쪽을 봄** 확인.

## v0.33 (2026-07-15) — 적 방향 버그 수정 (그런트가 옆/뒤를 보며 다가오던 문제)

**증상**: 그런트가 플레이어를 향해 오는데 몸은 엉뚱한 방향(옆/뒤)을 보고 있음.

**원인**
- v0.28에서 적 방향을 **이동 벡터**(`dir4(mvx,mvy)`)로 판정하도록 만들었음.
- 그런데 v0.26에서 그런트에 **선회(orbit) AI**를 넣었기 때문에, 그런트는 접근 중 대부분 옆으로 게걸음치거나 뒤로 물러남.
- 결과: 이동 방향을 따라 스프라이트가 옆/뒤를 보게 됨. 두 기능이 각각은 맞는데 조합에서 충돌.
- 측정: 교전 중 플레이어가 아래에 있는데 그런트가 `up`을 보는 프레임 확인.

**수정**
- 근접 적은 **발이 향하는 곳이 아니라 표적을 본다**는 원칙 적용.
- 그런트: 340px 이내 교전 중이면 항상 플레이어 방향으로 face 고정 (이동 방향 무시).
- shooter: 조준 중엔 `aimX/aimY`, 300px 이내 재배치 중엔 플레이어 방향 고정.
- 원거리에서 단순 접근할 때는 기존대로 이동 방향 사용 (선회를 안 하므로 문제 없음).
- charger는 기존 유지 — 돌진은 고정 벡터를 봐야 직진 관성이 살아있음.

**검증**: 그런트 45프레임 / shooter 45프레임 전부 방향 일치 (수정 전 3/30 불일치).

## v0.32 (2026-07-15) — 돌 던지는 고블린 (shooter) — 적 스프라이트 완성

**shooter → 돌 던지는 고블린으로 교체**
- 신규 4x3 시트 (512x384): 열 = down/up/left/right, 행 = walkA/walkB/throw.
- 던지기 윈드업 자세(3행) 정상 — 팔을 어깨 위로 젖히고 돌을 든 자세.
- 크기 0.78 스케일 / 렌더 높이 92 — 근접 고블린(96)보다 약간 작게, 빈약한 체형 반영.
- 걷기 사이클 가장 빠름 (`walkT/5`) — 종종거리는 느낌.
- 원본 3행 LEFT 칸은 손이 비어 있었으나, RIGHT 반전 대체 파이프라인 덕분에 자동 해결.

**조준 중 방향**
- shooter는 조준 중 거의 안 움직여서 이동 기반 방향 판정이 무의미 → `aimX/aimY` 고정 조준점 기준으로 방향 판정.
- 돌이 날아가는 방향과 던지는 자세가 일치.

**적 스프라이트 3종 완성**: grunt(고블린) / charger(오크) / shooter(돌 던지는 고블린) 전부 제미나이 생성 → 투명화 → 4방향 시트로 통합 완료.

**검증**: 3종 동시 스폰, 전 상태(approach/rush/windup/charge/rest/aim/cool) 정상, 돌 투척 확인, 에러 없음.

## v0.31 (2026-07-15) — 돌 투사체 (마법탄 → 던진 돌)

**투사체를 코드로 교체**
- 기존: 보라색 원 + 글로우 (`ctx.arc` 두 개) — 마법탄.
- 변경: `makeRock()` — 6~7각 불규칙 다각형을 스폰 시 1회 생성해서 고정. 매 프레임 회전(`spin += spinV`)시켜 공중에서 구르는 돌로 표현.
- 개체마다 다각형 형태·회전 방향·회전 속도가 랜덤 → 같은 돌이 두 번 안 나옴.
- 렌더: 바닥 그림자 타원 + 회색 돌 본체 + 어두운 외곽선 + 밝은 면 하나(입체감).
- 스프라이트 대신 코드로 간 이유: 반지름 5px라 그림으로 뽑아봐야 뭉개지고, 회전 애니메이션이 공짜로 나옴.

**조준 텔레그래프도 물리 투척으로**
- 기존: 보라색 점선 + 몸에서 커지는 보라 링 = 마법 차징 연출.
- 변경: 회색 점선 + **머리 위로 치켜든 돌**이 윈드업에 따라 커짐. 던지기 예비동작으로 읽힘.

**검증**: 돌 발사 확인 (7각형), spin 값 변화로 회전 확인, 런타임 에러 없음.

**남은 것**: shooter 스프라이트(돌 던지는 고블린) — 현재 아직 기존 보라색 도형.

## v0.30 (2026-07-15) — 오크 스프라이트 (charger)

**charger → 오크 스프라이트로 교체**
- 신규 4x3 시트 (512x384): 열 = down/up/left/right, 행 = walkA/walkB/charge.
- 고블린보다 크게 배치 (0.92 스케일, 렌더 높이 132 vs 고블린 96).
- 걷기 사이클을 더 느리게 (`walkT/8` vs 고블린 `/6`) — 육중한 느낌.

**대칭 디자인으로 재생성 (1차 폐기)**
- 1차 생성분은 한쪽 어깨 견갑 + 가슴 사선끈이 있었음. LEFT를 RIGHT 반전으로 만드는 파이프라인에서 **견갑이 반대쪽 어깨로 순간이동**하는 문제 발생. 정면(1열)과 뒷면(2열)의 견갑 위치도 서로 불일치.
- 원인은 내 프롬프트: "rusty iron shoulder pauldron"이라고 비대칭 요소를 직접 요청했음.
- 2차 프롬프트에서 견갑·사선끈·한쪽 팔보호대 등 좌우 비대칭 요소를 전부 금지하고 **이유(반전될 것이므로)까지 명시** → 대칭 디자인 확보.

**돌진 텔레그래프**
- windup 상태에서도 charge 자세(3행)를 표시. "온다!"는 예고가 되어 회피 타이밍을 읽을 수 있음.
- 돌진 중에는 방향을 `chgVx/chgVy` 고정 벡터로 판정 — 플레이어를 향해 몸이 돌아가지 않고 직진 관성 유지.
- 검증: windup → charge → rest 전 상태 확인, 돌진 중 방향 고정 확인.

**남은 것**: shooter(돌 던지는 고블린) + 돌 투사체.

## v0.29 (2026-07-15) — BGM 추가 (Skybound Caravan)

**음원 처리**
- 원본 6.15MB / 200kbps / 4분 18초 → **96kbps로 재인코딩 (2.9MB, 절반 이하)**. 게임 BGM엔 200kbps가 과함.
- 원본 끝부분이 페이드아웃돼 있어서 그냥 loop하면 소리가 사라졌다 갑자기 커지는 이음새가 생김 (끝 -38.6dB vs 시작 -27.5dB).
  → 페이드아웃 꼬리를 잘라내고(254초) 짧은 페이드인만 유지 → 이음새 -24.9dB / -29.8dB로 정합.

**임베드 방식 (중요)**
- base64로 index.html에 박지 **않음**. 6MB가 base64로 8MB가 되고, v0.27에서 HTML을 network-first로 바꿨기 때문에 **매 로드마다 8MB를 새로 받게 됨** — 모바일에서 치명적.
- 대신 `bgm.mp3` 별도 파일 + 서비스워커 정적 캐시(cache-first)에 등록. 첫 로드 후엔 오프라인에서도 재생되고 재다운로드 없음.

**재생 로직**
- 브라우저 자동재생 정책 때문에 로드 시 `play()`는 실패함 → 첫 탭/키 입력(user gesture)에서 재생 시작.
- 우상단 ♪ 버튼으로 on/off. 설정은 `localStorage`에 저장되어 다음 실행에도 유지.
- 탭/앱이 백그라운드로 가면 자동 일시정지, 돌아오면 재개.
- 볼륨 0.45 기본.

## v0.27 (2026-07-15) — 서비스워커 캐시 버그 수정 (버전이 안 올라가던 문제)

**증상**: 레포에는 v0.26이 올라가 있는데 브라우저/앱에서는 계속 v0.25가 뜸.

**원인 (내 설계 실수)**
- `sw.js`의 fetch 핸들러가 **cache-first**였음: `caches.match(req).then(hit => hit || fetch(req))`.
- 게임 전체가 `index.html` 한 파일에 들어있는데, 캐시에 index.html이 있으면 네트워크를 아예 조회하지 않음.
- 캐시 이름(`ys-v0_26`)을 올려도 브라우저가 **sw.js 자체를 캐시에서** 받으면 새 워커가 설치되지 않아 영원히 구버전에 잠김.
- 즉 "캐시 이름만 올리면 갱신된다"는 건 틀렸음.

**수정**
- HTML은 **network-first**로 전환: 항상 네트워크를 먼저 시도하고(`cache: 'no-store'`), 실패할 때만 캐시로 폴백. 오프라인 동작은 유지.
- 아이콘·매니페스트 등 정적 자산은 cache-first 유지 (자주 안 바뀜).
- 페이지 쪽에 자동 갱신 로직 추가: 로드 시 + 60초마다 `reg.update()`, 새 워커 감지되면 `SKIP_WAITING` → `controllerchange`에서 1회 리로드.

**결과**: 앞으로는 새로고침만 하면 최신 빌드가 뜸. 앱 삭제/재설치 불필요.

## v0.24 (2026-07-15) — 감속 중 걷기 자세 유지

**walk → idle 전환 부드럽게**
- 원인: 상태 선택이 `player._moving`(입력 유지 여부)를 봤음. 키를 떼는 순간 아직 관성으로 미끄러지는 중인데도 즉시 idle 자세로 바뀜 — 발이 멈췄는데 몸이 미끄러지는 부자연스러움.
- 수정: 입력이 아니라 **실제 속도**(`IDLE_SPD = 0.6`)를 기준으로 판정. 감속 중에는 walk 유지, 거의 멈췄을 때만 idle.
- 검증: 키 릴리즈 후 속도 3.9 → 0.88까지 walk 유지, 0.42부터 idle 전환.

**숨쉬기 위상 리셋**
- 이동/공격/대시 중에는 `breath = 0`으로 리셋. 멈출 때 호흡 중간부터 튀어나오지 않고 항상 새 호흡으로 시작.

## v0.22 (2026-07-15) — 주인공 행동 자연스러움 개선

**이동에 관성 추가**

- 결과: 출발 시 3.7 → 5.0 램프업, 정지 시 2.4 → 0 활강.

**방향 전환 보간**
- 기존: 입력 즉시 facing 벡터가 스냅 — 로봇처럼 꺾임.
- 변경: `fang`(각도) 상태 추가, 최단 회전 방향으로 프레임당 `TURN_RATE 0.35` 보간.
- 결과: 0 → -1.57 전환이 4프레임에 걸쳐 부드럽게.

**공격 중 전진 (lunge)**
- 기존: 스윙 중 완전 정지 (`atkT>0`이면 이동 코드 자체를 건너뜀).
- 변경: 액티브 프레임 동안 facing 방향으로 전진력 추가 (일반 1.5, 대시공격 3.2). 스윙 중 속도는 0.80씩 감쇠.
- 결과: 칼을 휘두르며 몸이 따라 들어감. 이스 특유의 "몸통 박치기" 느낌에 근접.

**대시 감속 곡선**
- 기존: `DASH_SPD` 고정 속도로 11프레임 슬라이드 — 얼음판 위를 미끄러지는 느낌.
- 변경: `s = DASH_SPD * (0.45 + 0.55*t²)` — 초반 폭발, 후반 감속. 대시 종료 시 잔여 속도 50% 캐리.

**대시 방향 즉시 확정**
- 대시는 turn 보간을 무시하고 현재 입력 방향으로 즉시 스냅. 회전 지연 때문에 원치 않는 방향으로 대시하는 문제 방지.

**테스트**
- Playwright 헤드리스 회귀 테스트로 4개 동작 모두 검증 (가속 램프 / 감속 활강 / 방향 보간 / 대시 테이퍼).
- JS 문법 검사 통과, 런타임 에러 없음.

### 튜닝 상수 (index.html)
```

DECEL     = 0.22   // 낮을수록 미끄러짐
```
